# MUSIT-LiDAR

Object detection and tracking from a stationary Velodyne VLP-16 LiDAR
recording: decode the raw sensor capture, detect objects by clustering,
track them across the sensor's rotations, and plot detected objects on a
real-world map.

For the methodology behind each step (why RANSAC, why these four clustering
methods, how tracking/georeferencing work, and known limitations), see
[REPORT.md](REPORT.md).

## The one thing to know before you touch this code

**Clustering and tracking must run on the LOCAL point cloud** (x, y, z in
meters relative to the sensor), **never on a georeferenced one.**
Georeferenced coordinates are UTM easting/northing (huge numbers, ~300000+)
stored as float32, which loses ~0.5m of precision at that magnitude — enough
to corrupt small object bounding boxes. The correct order is: cluster/track
first on the local cloud, then georeference just the detected objects'
centroids and boxes afterwards for mapping. `main.py` already does this in
the right order — see [Pipeline stages](#pipeline-stages) below.

## Folder structure

```
MUSIT-LiDAR/
├── main.py                 # pipeline driver: read -> fit ground -> cluster -> track -> georeference results
├── src/
│   ├── lidar_io.py                  # shared PLY/CSV I/O, frame-index derivation
│   ├── ground.py                    # shared ground-plane fit/split + save/load ground_plane.json
│   ├── read_lidar_data.py           # stage 1: decode .pcap -> local point cloud (.ply)
│   ├── fit_ground_plane.py          # stage 2: RANSAC ground plane, fit ONCE -> ground_plane.json
│   ├── cluster_objects.py           # stage 3: clustering (reuses ground_plane.json) -> detected objects
│   ├── track_objects.py             # stage 4: per-frame clustering + tracking across rotations
│   ├── georeference_point_cloud.py  # standalone utility: local cloud -> UTM (for GIS tools)
│   ├── plot_clusters_on_map.py      # stage 5: georeference detected objects -> OSM map
│   └── plot_point_cloud.py          # utility: quick top-down/side-view plots of any point cloud
├── notebooks/
│   └── plot_objects_on_map.ipynb    # interactive version of stage 5, for calibrating heading/offset
├── data/                    # raw sensor captures (.pcap)
└── outputs/                 # everything generated by the pipeline (point clouds, CSVs, plots)
```

## Quick start: the full pipeline

```
python main.py data/marina-giannis-test2-20260513-133011.pcap \
    --lat 37.430785059983215 --lon 24.941538092360716 --heading 280 \
    --output-dir outputs/
```

This runs, in order:
1. **read** — decodes the `.pcap` into `outputs/point_cloud.ply` (local frame).
2. **fit ground** — RANSAC ground-plane fit, done exactly **once** -> `outputs/ground_plane.json`. This is the slow step (hundreds of RANSAC iterations over the whole cloud); fitting it once and sharing it with both stages below avoids repeating that cost per clustering method or per `track_objects.py` run.
3. **cluster** — clustering on the local cloud using the shared ground plane, one snapshot -> `outputs/detected_objects.csv`/`.png`.
4. **track** — clusters every rotation (same shared ground plane) and links clusters into tracks across frames -> `outputs/tracks_*.csv`/`.png`.
5. **georeference results** — projects the detected objects (not the raw cloud) onto an OSM basemap -> `outputs/clusters_map.png`. Needs internet access for map tiles; skipped automatically (with a warning) if `--lat`/`--lon`/`--heading` aren't given, or if it fails.

A stage is skipped if its output file already exists (reading the `.pcap`,
fitting the ground plane, and clustering the full multi-rotation cloud can
each take several minutes) — pass `--force` to re-run everything anyway. Run
`python main.py --help` for every flag `main.py` exposes; for full control
over one stage, run that stage's script directly (see below).

## Pipeline stages, standalone

Each stage is a full CLI script with its own flags — `python src/<script>.py --help`
for the complete list, or see the docstring at the top of each file.

```
# 1. read: .pcap -> local point cloud
python src/read_lidar_data.py data/capture.pcap outputs/point_cloud.ply

# 2. fit ground plane ONCE -- reuse it below instead of re-running RANSAC per method/per script
python src/fit_ground_plane.py outputs/point_cloud.ply --output outputs/ground_plane.json

# 3. cluster: one snapshot, compare clustering methods against each other (no RANSAC refit)
python src/cluster_objects.py outputs/point_cloud.ply --ground-plane outputs/ground_plane.json --method dbscan --output-prefix outputs/dbscan_
python src/cluster_objects.py outputs/point_cloud.ply --ground-plane outputs/ground_plane.json --method hdbscan --output-prefix outputs/hdbscan_
python src/cluster_objects.py outputs/point_cloud.ply --ground-plane outputs/ground_plane.json --method voxel-cc --output-prefix outputs/cc_
python src/cluster_objects.py outputs/point_cloud.ply --ground-plane outputs/ground_plane.json --method euclidean --output-prefix outputs/euclidean_

# 4. track: same object, linked across rotations, classified static/moving
# (track_objects.py's own filenames already start with "tracks_", so just point
# --output-prefix at the directory, the same way cluster_objects.py's is used above)
python src/track_objects.py outputs/point_cloud.ply --ground-plane outputs/ground_plane.json --output-prefix outputs/

# 5. georeference results: detected objects -> real-world map
python src/plot_clusters_on_map.py outputs/detected_objects.csv \
    --lat 37.430785059983215 --lon 24.941538092360716 --heading 280 \
    --point-cloud outputs/point_cloud.ply --output outputs/clusters_map.png

# optional utility: georeference the WHOLE point cloud (for GIS tools, not for clustering/tracking)
python src/georeference_point_cloud.py outputs/point_cloud.ply \
    --lat 37.430785059983215 --lon 24.941538092360716 --heading 280 \
    --output outputs/geo_v1.ply

# optional utility: quick top-down/side-view plots of any point cloud
python src/plot_point_cloud.py outputs/point_cloud.ply --output-prefix outputs/local_
```

## Notebook: interactive map plotting

`notebooks/plot_objects_on_map.ipynb` is the interactive version of stage 5.
Georeferencing a LiDAR scan onto a map usually needs a bit of trial and error
on `heading`/`heading_offset` until the objects line up with real features on
the basemap — the notebook lets you tweak the parameters cell and re-run
just the plotting cell, instead of re-running the CLI script and re-opening
a PNG each time.
